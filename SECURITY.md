5. После исправления будет опубликован CVE (если применимо)

## 🔐 Управление секретами

### Хранение API ключей

| Тип ключа | Где хранить | Как хранить |
|-----------|-------------|-------------|
| Telegram Bot Token | `.env` | Переменная `TELEGRAM_BOT_TOKEN` |
| ProxyAPI API Key | `.env` | Переменная `PROXYAPI_API_KEY` |
| GigaChat Key | `.env` | Переменная `GIGACHAT_AUTHORIZATION_KEY` |
| OpenAI API Key | `.env` | Переменная `OPENAI_API_KEY` |

**КРИТИЧЕСКИ ВАЖНО:**
- ✅ Файл `.env` **всегда** должен быть в `.gitignore`
- ✅ Используйте `.env.example` как шаблон с плейсхолдерами
- ❌ **НИКОГДА** не коммитьте реальные ключи в Git
- ❌ Не делитесь ключами в чатах, почте, коде

### Проверка на случайно закоммиченные секреты

**Ручная проверка:**
```bash
# Проверить, не закоммичен ли .env
git ls-files .env

# Проверить историю на секреты
git log -p --all -- .env | grep -E "sk-|token|key"
```

**Автоматическая проверка (перед push):**
```powershell
.\scripts\check_before_push.ps1
```

**Сканирование безопасности:**
```powershell
.\scripts\security_scan.ps1
```

## 🔄 Ротация ключей

### Рекомендуемая частота ротации

| Тип ключа | Частота | Примечание |
|-----------|---------|------------|
| Telegram Bot Token | 180 дней | Или при компрометации |
| ProxyAPI API Key | 90 дней | Ключ виден только 1 раз при создании |
| GigaChat Key | 90 дней | |
| OpenAI API Key | 90 дней | |

### Немедленная ротация требуется при:
- Утере доступа к системе
- Подозрении на утечку
- Смене сотрудников с доступом
- Инцидентах безопасности

### Процедура ротации:

1. **Сгенерировать новый ключ** в консоли провайдера
2. **Обновить `.env`** с новым ключом
3. **Проверить работоспособность:**
   ```bash
   python -c "from config import Settings; s = Settings.from_env(); print('✅ Config OK')"
   ```
4. **Перезапустить бота**
5. **Отзвать старый ключ** (после подтверждения работы)

**Скрипт помощи:**
```powershell
.\scripts\rotate_secrets.ps1
```

## 🌐 Сетевая безопасность

### HTTPS/TLS

- ✅ Все API вызовы используют HTTPS
- ✅ SSL верификация **включена по умолчанию**
- ⚠️ Для самоподписанных сертификатов в разработке можно использовать `verify_ssl=False`

**В production всегда используйте:**
```python
verify_ssl=True
```

### Проxy-серверы

При использовании корпоративных прокси:
```env
# Опционально: прокси для ProxyAPI
PROXYAPI_PROXY_URL=http://proxy.company.com:8080
```

**Рекомендации:**
- Используйте выделенные прокси для production
- Настройте аутентификацию для прокси
- Логирование трафика через прокси

## 📁 Безопасность данных

### Векторная база (ChromaDB)

**Хранение:**
```env
CHROMA_PERSIST_DIR=./chroma_db
```

**Рекомендации для production:**
- ✅ Шифрование диска для `chroma_db/`
- ✅ Ограничение доступа к файловой системе
- ✅ Регулярные бэкапы
- ❌ Не хранить в общедоступных облачных хранилищах без шифрования

### Пользовательские данные

**Хранение:**
```env
# По умолчанию: ./user_data.json
```

**Рекомендации:**
- ✅ Шифрование `user_data.json`
- ✅ Ограничение прав доступа (chmod 600)
- ✅ Регулярные бэкапы

## 🛡️ Рекомендации по безопасности

### Для разработчиков

1. **Перед коммитом:**
   ```powershell
   .\scripts\check_before_push.ps1
   ```

2. **При добавлении новых ключей:**
   - Добавить переменную в `.env.example`
   - Задокументировать в `SECURITY.md`
   - Добавить в `.gitignore` если нужно

3. **При работе с секретами:**
   - Использовать переменные окружения
   - Не логировать ключи
   - Валидировать входные данные

### Для production

1. **Использовать секрет-менеджер:**
   - HashiCorp Vault
   - AWS Secrets Manager
   - Azure Key Vault

2. **Настроить мониторинг:**
   - Логирование доступа к секретам
   - Уведомления о подозрительной активности
   - Мониторинг использования API ключей

3. **CI/CD безопасность:**
   - Сканирование на секреты в pipeline
   - Использование OIDC для аутентификации
   - Минимальные права для CI/CD
